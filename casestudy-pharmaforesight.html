Focussing on Business Results

See more

For most of our projects, matching the brief isn’t going to get a clients’ project over the finish line, so we started by focussing on what “success” looks like for this project.

By understanding that the purpose of this tool is to show off their AI algorithm to potential clients during a Demo Meeting, where someone will be describing and operating it in-person, we understood which parts of the brief should be challenged (for example, having the tool on 2 pages).

We were also able to recommend using query parameters in the URL, which means they can click one of many pre-made links during the meeting, and voila! All the drug names, NLM identifiers, etc are pre-populated, ready to press the big exciting “Predict” button.

Engineering Value

See more

Startups depend on being resourceful, and we found an opportunity to collaborate with a business-savvy & tech-savvy client to engineer some serious value in a limited budget.

By developing API blueprints, defining each persons’ responsibility upfront, and being clear the project’s requirements (in terms of skills and experience), we were able to appoint someone who could bring his service design experience, visual design eye, and then write clean React.js code for a snappy frontend-only single page app (SPA). It takes a good project plan, and a the right contractor to be able to deliver a project at this level.

When speed matters

See more

When our client has a Demo Meeting that’s under a month away, we’ll see if we can rise to the challenge. For this project, Calibre’s Director and PharmaForesight’s Founder came together and planned the project in a 1hr meeting, and email communication was kept concise, clear, and complete. By the time we kicked the project off, Joshua had all the information + assets at the ready, and was able to start crafting.

To avoid slowing him down, we agreed to treat feedback as “suggestions” to Joshua, and let him have final say over how he met our brief. All of these efforts paid off when the project was ready to demo in under 2 weeks.

CALIBRE TIP

When managing frontend and backend work at the same time, the best way to clarify the responsibility of each party is by writing API documentation upfront. By defining this “interface”, you can avoid huge delays in a project, where both developers could be emailing back and forth requesting changes to each others’ work in order for their own codebase to function correctly. We use postman for mock APIs.

p.s. Generally, this API documentation should be written by the frontend developer.
